# Claude Doctor

> Структурные предохранители для Claude Code. Хук-напоминания в момент действия — для прод-операций, архитектурных вопросов и заявлений о завершённости без подтверждения.

[🇬🇧 Read in English](README.md)

**Зачем это нужно.** Правила в CLAUDE.md и AGENTS.md Claude читает в начале сессии и к середине дрейфует от них — это описано в отчётах на anthropics/claude-code (#37818, #36492, #29564, #37297), об этом пишет Andrej Karpathy, это подтверждает независимое академическое исследование Huang et al. 2023. Claude Doctor не добавляет ещё одно правило — он инжектит напоминание ровно в тот момент, когда паттерн вот-вот повторится (`UserPromptSubmit` / `Stop`). Полная аргументация с цитатами — в [references/philosophy.md](references/philosophy.md).

**Статус:** v0.1.0, детекция на Stop-хуке в режиме только-лог (настраивается). Кроссплатформенный — Linux, macOS, Windows с Git Bash.

## Что делает

**Детектор прод-ключевых слов** (UserPromptSubmit). Срабатывает, когда в сообщении пользователя есть слова из категории прод-операций — дефолты двуязычные (`deploy`, `migrate`, `publish to`, `push to production` + русские эквиваленты `деплой`, `миграци`, `опублик`). При срабатывании инжектит блок самопроверки из пяти пунктов, который модель обязана заполнить конкретными tool output'ами до любого действия. Расширяется через `prod_keywords_add` в проектном конфиге.

**Детектор архитектурных вопросов** (UserPromptSubmit). Срабатывает на советные формулировки (`how should I...`, `what's the best approach`, `посоветуй`, `как лучше`, `какой подход`). При срабатывании требует как минимум один tool call на чтение реальных файлов (Read / Bash / Grep / Glob) до генерации любой рекомендации. Закрывает Паттерн B из документа о философии — на советно-сформулированных вопросах модель склонна отвечать из auto-loaded контекста, а не из свежего чтения файлов.

**Детектор фабрикаций** (Stop). Сканирует последний ответ модели перед её остановкой. Две проверки: (1) фабрикация атрибуции — модель приписывает пользователю «кодовые слова», которых пользователь на самом деле не произносил в утвердительной форме; (2) заявления о завершённости без доказательств — модель говорит «готово», «работает», «задеплоено» в ответе, где не было ни одного инструмента, подтверждающего реальное состояние (Read, Bash, Grep, Glob, WebFetch, MCP-чтения). В v0.1 только-лог — флаги пишутся в `$CLAUDE_PLUGIN_DATA/audit.log` (fallback: `~/.claude/plugin-data/claude-doctor/audit.log`).

**SessionStart-анализатор**. Раз в сессию читает audit log за последние семь дней, строит сводку, пишет её в человекочитаемый файл мониторинга и инжектит однострочный статус в начальный контекст сессии. Делает накопленный аудит видимым без ручного открытия файлов.

## Установка

Эти три команды — слэш-команды, которые ты пишешь **внутри Claude Code** (в том же поле для ввода, где обычно пишешь сообщения). Это не команды терминала — не вставляй их в bash или PowerShell.

Шаг 1 — зарегистрировать marketplace:

```
/plugin marketplace add alenazaharovaux/claude-doctor
```

Ожидаемый ответ Claude Code: подтверждение что marketplace добавлен, со списком одного доступного плагина (`claude-doctor`).

Шаг 2 — установить плагин:

```
/plugin install claude-doctor@claude-doctor
```

Ожидаемый ответ: плагин появляется во вкладке «Installed» меню `/plugin`.

Шаг 3 — создать конфиг для проекта:

```
/claude-doctor:setup
```

Эта команда создаёт файл `.claude/claude-doctor.local.md` внутри текущей директории проекта (той папки, в которой запущен Claude Code). В файле — все доступные настройки с дефолтными значениями. Открой его в любом текстовом редакторе и подкрути под себя.

Хуки работают и без конфиг-файла (используют встроенные дефолты), но файл нужен, если хочешь добавить специфичные для проекта ключевые слова, сменить язык или выключить отдельные детекторы.

### Требования

**Python 3.** Открой терминал и запусти:

```bash
python3 --version
# или, если выдаёт "command not found":
python --version
```

Если видишь что-то вроде `Python 3.10.0` или новее — всё нормально. Если обе команды говорят «command not found», установи Python отсюда: [python.org/downloads](https://www.python.org/downloads/) (любая версия 3.x подойдёт, дополнительные пакеты не нужны).

**На Windows — Git for Windows.** Плагин использует маленький shell-скрипт (`hooks/run.sh`) для авто-детекта Python. На Windows этому скрипту нужен `sh`, который идёт в комплекте с Git Bash. Скачай и установи Git for Windows здесь: [git-scm.com/download/win](https://git-scm.com/download/win). Опции установки оставь по умолчанию — Git Bash добавится автоматически.

Чтобы проверить установку, открой любой терминал (PowerShell или cmd) и запусти `sh --version`. Если видишь версию Bash — всё на месте.

## Первый тест после установки

Чтобы убедиться, что плагин работает, отправь Claude три сообщения по одному (как отдельные промпты) и посмотри на ответы.

**Тест 1 — детектор прод-ключевых слов.** Напиши на любом из языков (по умолчанию плагин ловит и английский, и русский):

> let's deploy this to production

или

> давай задеплоим это на прод

Ожидание: до того, как Claude ответит по сути, ты увидишь блок, начинающийся с `🔍 CLAUDE DOCTOR — production keyword detected` (или, если у тебя в конфиге `language: "ru"`, на русском), и чек-лист самопроверки из пяти пунктов. Если видишь этот блок, хук `prod-keyword-detector` срабатывает.

**Тест 2 — детектор архитектурных вопросов.** Начни новый промпт и напиши:

> how should i structure this project?

или

> посоветуй, как лучше организовать структуру проекта

Ожидание: блок, начинающийся с `🔍 CLAUDE DOCTOR — architectural/advisory question detected`, и инструкции, требующие от Claude прочитать реальные файлы до того, как он ответит. Если видишь — хук `architectural-question-detector` работает.

**Тест 3 — SessionStart-сводка.** Полностью выйди из Claude Code (закрой терминал или выполни `/quit`), затем запусти новую сессию в любом проекте. В первых системных сообщениях новой сессии посмотри на строку вида:

```
📊 Claude Doctor: 0 flags in last 7 days. Good or hooks disabled.  (details: ...путь...)
```

Если видишь — хук `SessionStart` работает. Путь в скобках указывает на файл мониторинга, который пишется на старте каждой сессии.

**Детектор фабрикаций сложнее проверить напрямую** — это `Stop`-хук, он срабатывает, когда Claude заканчивает ход, и пишет в лог в фоне. Чтобы убедиться, что хук хоть раз запустился, проверь heartbeat-файл в терминале:

```bash
cat ~/.claude/plugin-data/claude-doctor/heartbeat.log
```

Каждая строка — один `Stop`-event с timestamp и session ID. Если файл существует и в нём есть строки, хук работает.

Если какой-то тест не проходит — смотри [Troubleshooting](#troubleshooting) ниже.

## Интерактивный triage флагов

Интерактивный поток использует `AskUserQuestion` — встроенный инструмент Claude Code, который отрисовывает кликабельные кнопки прямо в чате. Ничего дополнительно ставить не нужно: если у тебя запущен Claude Code — этот инструмент уже доступен. Официальная справка по slash-командам и инструментам: [code.claude.com/docs/en/slash-commands](https://code.claude.com/docs/en/slash-commands).

Через несколько дней работы в audit-логе накапливаются флаги. Часть из них — настоящие моменты фабрикации, когда Claude заявил о выполнении без проверки. Часть — шум: фраза, которая просто встречается в твоём стиле, без желания ввести в заблуждение. Лог сам по себе эти случаи не различает; делаешь это ты — через `/claude-doctor:triage`.

Команда читает лог, отбрасывает флаги, уже обработанные в предыдущих запусках, и проводит тебя по оставшимся. Для самых частых фраз сначала показывается массовое решение — кнопки `AskUserQuestion` позволяют одним кликом разобраться со всеми вхождениями:

- **Block all** — добавляет фразу в `claim_phrases_blocking`. Следующий раз, когда Claude напишет её без вызова инструмента-доказательства (`Read`, `Bash`, `WebFetch` и так далее), хук `Stop` прервёт ход с кодом 2 и потребует либо проверить утверждение, либо отозвать его.
- **Ignore all** — добавляет фразу в `claim_phrases_ignore`. Будущие вхождения отсеиваются до того, как попадут в лог. Это решение для фраз, которые естественно входят в твою речь.
- **Review individually** — разворачивается в цикл по каждому отдельному флагу с полным контекстом, чтобы решать по случаю.
- **Skip** — оставляет флаг на потом.

После того как частотные фразы обработаны массово, остальные показываются по одному с теми же тремя опциями без «all». Фразы, которые ты заблокировала или проигнорировала, автоматически фильтруются при каждом запуске — следующий `/triage` не будет спрашивать про уже решённое. Фразы, которые ты пропустила (skip), вернутся в следующий запуск — skip это «решу позже», а не «больше не показывай».

Смысл такого потока — постепенное обучение. Сначала все фразы обрабатываются одинаково; за несколько сессий блокирующий список собирает те немногие паттерны, которые действительно предсказывают проблему у тебя в работе, а игнор-список поглощает пересечения с обычной письменной речью. Когда оба списка стабилизируются, детектор работает только на остатке — там, где реально бывает сигнал.

Если хочется работать руками, открой `.claude/claude-doctor.local.md` и редактируй `claim_phrases_blocking` или `claim_phrases_ignore` напрямую. `/triage` — это удобство, не обязательство.

## Конфигурация

Редактируй `.claude/claude-doctor.local.md` в своём проекте:

| Поле | Тип | Дефолт | Что делает |
|---|---|---|---|
| `enabled` | bool | `true` | Главный выключатель. При `false` все хуки плагина пропускаются. |
| `language` | string | `"en"` | Язык инжекта: `"en"`, `"ru"` или `"both"` |
| `prod_keywords_add` | list[str] | `[]` | Слова, добавленные к дефолтному списку prod-триггеров |
| `prod_keywords_replace` | list[str] | `[]` | Если не пустой — полностью заменяет дефолты |
| `architectural_enabled` | bool | `true` | Включить детектор архитектурных вопросов |
| `fabrication_enabled` | bool | `true` | Включить детектор фабрикаций на Stop-хуке |
| `claim_phrases_add` | list[str] | `[]` | Дополнительные completion-claim фразы |
| `claim_phrases_replace` | list[str] | `[]` | Если не пустой — полностью заменяет дефолты |
| `claim_phrases_blocking` | list[str] | `[]` | v0.2+: фразы, которые включают `exit 2` на Stop-хуке, если замечены без инструмента-доказательства. Обычно заполняется через `/claude-doctor:triage`. |
| `claim_phrases_ignore` | list[str] | `[]` | v0.2+: фразы, полностью исключённые из детекции. Заполняется через `/claude-doctor:triage` (Ignore / Ignore all). |
| `last_triage_timestamp` | string | `""` | Legacy-поле с v0.2.0. Оставлено в схеме для обратной совместимости, но с v0.2.1 не используется. Skip-фразы теперь возвращаются в каждый triage до ignore/block. |
| `scan_history` | bool | `true` | Разрешить детектору читать `~/.claude/history.jsonl` |
| `monitoring_path` | string | `""` | Переопределить путь к файлу мониторинга |

Пример:

```yaml
---
enabled: true
language: "both"
prod_keywords_add: ["kubectl apply", "terraform apply", "helm upgrade"]
architectural_enabled: true
fabrication_enabled: true
scan_history: false
monitoring_path: "~/notes/claude-audit.md"
---

# Заметки (произвольные, плагин игнорирует тело)
Настроено для инфраструктурного проекта — добавлены K8s/TF/Helm глаголы в прод-триггеры.
History-scan выключен по причинам приватности.
```

Изменения применяются со **следующего** вызова хука. Рестарт Claude Code не требуется — конфиг читается на каждый event.

## Философия

Почему это хук, а не ещё одно правило в CLAUDE.md? Коротко: правила в CLAUDE.md активируются постфактум, хуки активируются в моменте. Подробно, с внешними цитатами (четыре issue на anthropics/claude-code, публичные комментарии Andrej Karpathy, Huang et al. 2023 про self-correction у LLM, исследование Anthropic про alignment-faking) — в [references/philosophy.md](references/philosophy.md).

## Troubleshooting

**Хуки молчат — ничего не происходит на триггерных словах.** Запусти `claude --debug` и посмотри в выводе сообщения о регистрации хуков. Если хуки не зарегистрированы — проверь, что плагин включён в меню `/plugin`. Если зарегистрированы, но не срабатывают — проверь, что `python3` или `python` доступны в `PATH`.

**Windows: `sh: command not found`.** Лаунчер плагина (`hooks/run.sh`) требует доступный `sh`. На Windows это даёт Git Bash. Установи [Git for Windows](https://git-scm.com/download/win) — Git Bash идёт в комплекте и автоматически добавляет `sh` в `PATH`.

**Изменения конфига не применяются.** Конфиг перечитывается на каждый вызов хука, но ты должен триггернуть event, чтобы увидеть новое поведение. Отправь новое сообщение с триггерным словом. Если всё равно не работает — проверь, что файл сохранён и что YAML-frontmatter синтаксически корректен (начинается и заканчивается отдельными строками `---`).

**`/plugin update` говорит «already at latest» хотя на GitHub новый релиз.** Известный баг Claude Code — см. [anthropics/claude-code issue #25244](https://github.com/anthropics/claude-code/issues/25244). Plugin manager не делает `git pull` локального клона marketplace перед проверкой версии. Workaround:

```bash
cd ~/.claude/plugins/marketplaces/claude-doctor && git pull
# затем в Claude Code:
/plugin update claude-doctor@claude-doctor
```

**Где логи?** Audit log: `$CLAUDE_PLUGIN_DATA/audit.log`, если эта env var установлена Claude Code, иначе `~/.claude/plugin-data/claude-doctor/audit.log`. Heartbeat log (доказывает, что хук запускался): та же директория, `heartbeat.log`. Файл мониторинга: та же директория, `monitoring.md`, если не переопределён через `monitoring_path`.

**Как выключить без удаления?** Два варианта. (1) `/plugin disable claude-doctor@claude-doctor` — отключает глобально до ручного включения. (2) В `.claude/claude-doctor.local.md` поставь `enabled: false` — отключает для конкретного проекта. Второй вариант предпочтительнее, если хочешь чтобы плагин работал везде, кроме одного конкретного репо.

**Ложные срабатывания fabrication-detector.** В v0.1 детектор работает в режиме только-лог, так что ложные срабатывания не блокируют работу — они накапливаются в audit log. Просматривай лог, подстраивай `claim_phrases_replace`, убирая фразы, которые дают шум на твоём стиле письма. Систематические паттерны ложных срабатываний стоит репортить issue'ом.
